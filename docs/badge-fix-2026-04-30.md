# 앱 아이콘 배지 카운트 버그 수정 (2026-04-30)

## 증상

삼성 기기에서 실제 읽지 않은 메시지가 3~5개인데 배지에 8, 10, 15 등 부풀린 숫자가 표시되는 문제.

---

## 원인 분석 및 수정 내역

### 1차 수정 — FCM 중복 배달로 인한 배지 카운트 부풀림

**원인:**  
삼성 Doze Mode 특성상 FCM 메시지가 중복으로 배달되는 경우가 있음.  
기존 코드는 FCM 수신 시마다 `_incrementBadge()`(로컬 카운터 +1)를 호출하고 있었기 때문에,  
1개 메시지가 3회 배달되면 배지가 3씩 증가하는 문제가 발생.

**수정:**  
`_incrementBadge()` 대신 서버의 `/api/chats/unread-count` API를 호출해 실제 미읽음 수를 받아와 배지에 덮어쓰는 방식으로 변경.

```dart
// fcm_service.dart — 추가된 함수
Future<int> _fetchAndStoreChatCount(SharedPreferences prefs, String token) async {
  try {
    final response = await http.get(
      Uri.parse('${ApiConfig.baseUrl}/api/chats/unread-count'),
      headers: {'Authorization': 'Bearer $token'},
    );
    if (response.statusCode == 200) {
      final decoded = jsonDecode(utf8.decode(response.bodyBytes));
      if (decoded is Map && decoded['data'] is num) {
        await prefs.setInt(_kBadgeChatKey, (decoded['data'] as num).toInt());
      }
    }
  } catch (_) {}
  return _readTotalBadge(prefs);
}
```

백그라운드/포그라운드 FCM 핸들러 모두에서 채팅 알림은 `_incrementBadge` → `_fetchAndStoreChatCount`로 교체.

**커밋:** `79b2d62`

### 2차 수정 — 백그라운드 상태에서 STOMP chatRead 호출로 배지 0 초기화

**원인:**  
1차 수정 후, 앱이 백그라운드에 있는 동안 파트너 메시지가 STOMP로 수신되면  
`chat_page.dart`의 STOMP 핸들러가 `POST /api/chat/read`(chatRead)를 무조건 호출.  
→ 서버에서 `lastReadChatId`가 최신 메시지 ID로 갱신  
→ `_fetchAndStoreChatCount` 호출 시 미읽음 수 = 0 반환  
→ `_showNotif(total: 0)` 호출 → 삼성이 `number: 0` 알림을 받아 알림 트레이 전체 소거

**수정:**  
STOMP 핸들러에서 chatRead 호출 시 `AppLifecycleState.resumed` 상태인 경우에만 호출하도록 조건 추가.

**커밋:** `86af05a`

---

### 3차 수정 — HOME 버튼 종료 시 lastReadChatId 미갱신

**원인:**  
`lastReadChatId` 갱신(chatRead)이 `initState`에서만 발생.  
사용자가 채팅을 읽고 HOME 버튼으로 나가면 `dispose`가 호출되지 않고  
`AppLifecycleState.paused`만 발생 → chatRead 미호출 → `lastReadChatId` 미갱신.  
이후 FCM 수신 시 서버가 이미 읽은 메시지들을 미읽음으로 계산해 배지 부풀림 재발.

**DB 증거:**  
사용자가 모두 읽은 상태임에도 `lastReadChatId = 3601`, `MAX(id) = 3606` — 5개 차이.

**수정:**  
chatRead를 다음 모든 시점에서 호출하도록 확장:

| 호출 시점 | 이유 |
|---|---|
| `initState` | 채팅 진입 시 기존 메시지 커버 |
| STOMP (파트너 메시지 수신 시) | 채팅 중 실시간 갱신 |
| `AppLifecycleState.paused` | HOME 버튼 최선 시도 |
| `dispose` | 뒤로가기 / 탭 전환 시 (신뢰도 높음) |
| `AppLifecycleState.resumed` | 백그라운드→포그라운드 복귀 시 보정 |

```dart
// chat_page.dart
@override
void didChangeAppLifecycleState(AppLifecycleState state) {
  if (state == AppLifecycleState.resumed) {
    FcmService().clearChatNotifications();
    ApiClient.post(Uri.parse(ApiEndpoints.chatRead)).catchError((_) {});
    _fetchHistory();
    // WebSocket 재연결...
  } else if (state == AppLifecycleState.paused) {
    ApiClient.post(Uri.parse(ApiEndpoints.chatRead)).catchError((_) {});
  }
}

@override
void dispose() {
  WidgetsBinding.instance.removeObserver(this);
  FcmService().setChatActive(false);
  ApiClient.post(Uri.parse(ApiEndpoints.chatRead)).catchError((_) {});
  stompClient?.deactivate();
  // ...
}
```

**커밋:** `66d6f48`

### 4차 수정 — STOMP chatRead 제거 시도 (잘못된 방향)

**시도:**  
"어차피 `paused`/`dispose`/`resumed`가 chatRead를 커버하니 STOMP에서 매 메시지마다 호출할 필요 없다"는 판단으로 STOMP 핸들러의 chatRead 제거.

**문제:**  
삼성이 백그라운드 HTTP 요청(`paused` 시점의 chatRead)을 완료 전 프로세스를 종료하는 경우 발생.  
채팅창에서 메시지(3615, 3616)를 보내는 동안 파트너가 메시지(3617, 3618)를 보냄 → STOMP로 실시간 수신 → chatRead 없어 `lastReadChatId` 미갱신.  
이후 FCM으로 새 메시지(3620, 3621) 수신 → 서버 기준 미읽음: 3617, 3618, 3620, 3621 = 4개 → 배지 4.  
기대값은 3620, 3621에 해당하는 2개.

**커밋:** `cfe429a` (이후 바로 롤백)

### 5차 수정 — STOMP chatRead 복원

**수정:**  
STOMP 핸들러에 chatRead 복원. lifecycle 조건 없이 무조건 호출.  
(2차 수정의 lifecycle 가드 제거 — 현재는 `_fetchAndStoreChatCount`가 정확한 서버 값을 쓰기 때문에  
배경에서 chatRead → count=0 이 되어도 FCM 배지 표시 시점에 다시 서버 조회하므로 문제 없음)

**커밋:** `ea09a03`

## 최종 아키텍처 요약

### chatRead 호출 포인트 (`POST /api/chat/read`)

```
채팅창 진입 (initState)
  └─ 기존에 쌓인 미읽음 메시지 커버

STOMP 파트너 메시지 수신
  └─ 채팅 중 실시간 갱신 + 삼성 kill 보험

HOME 버튼 (AppLifecycleState.paused)
  └─ 최선 시도 (삼성이 완료 보장 안 함)

뒤로가기 / 탭 전환 (dispose)
  └─ 신뢰도 높음 — _pages[_selectedIndex] 방식이므로 탭 전환 시 dispose 호출됨

백그라운드→포그라운드 복귀 (AppLifecycleState.resumed)
  └─ paused 시 실패했을 경우 보정
```

### 배지 카운트 산출 흐름

```
FCM 수신
  └─ 채팅 알림인가?
       ├─ YES → _fetchAndStoreChatCount() — 서버 /api/chats/unread-count 조회 (truth source)
       └─ NO  → _incrementBadge() — 로컬 카운터 +1

서버 unread-count 계산
  └─ members.last_read_chat_id 기준
       └─ COUNT(id > lastReadChatId AND writer_uid != 나)
```

### 핵심 설계 원칙

- **배지 카운트는 로컬 증가 방식 사용 불가** — FCM 중복 배달(삼성 Doze) 때문
- **서버 truth-source 방식 필수** — `/api/chats/unread-count` API 직접 조회
- **chatRead는 여러 곳에 중복 배치** — 삼성 백그라운드 kill로 인해 단일 포인트 신뢰 불가
